    **1. Create migration `supabase/migrations/003_wishlist.sql`:**

    Create `wishlist_submissions` table:
    ```sql
    CREATE TABLE IF NOT EXISTS wishlist_submissions (
      id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
      name        TEXT NOT NULL,
      email       TEXT NOT NULL,
      company     TEXT,
      industry    TEXT,
      message     TEXT,
      created_at  TIMESTAMPTZ NOT NULL DEFAULT now()
    );
    ```

    Enable RLS on the table. Add a single policy allowing service_role to insert (all user-facing writes go through the API which uses service role client). No SELECT/UPDATE/DELETE policies for anonymous users — data is admin-only readable.

    ```sql
    ALTER TABLE wishlist_submissions ENABLE ROW LEVEL SECURITY;
    -- No user-facing policies. All access via service role in API endpoints.
    ```

    **2. Add Zod schema to `server/utils/validators.ts`:**

    Append to the existing file (do NOT modify existing schemas):
    ```typescript
    // --- Wishlist Schema ---
    export const wishlistSubmissionSchema = z.object({
      name: z.string().min(1, 'Nama wajib diisi').max(100),
      email: z.string().email('Email tidak valid'),
      company: z.string().max(100).optional(),
      industry: z.string().max(100).optional(),
      message: z.string().max(250).optional(),
    })
    ```

    The `industry` field accepts any string up to 100 chars because the "others" option allows free text input. Validation of allowed industry values is not enforced server-side — the client presents the dropdown with predefined options + free text "Lainnya" option.

    - Return `{ success: true }` with status 201
    - On validation error, return 400 with Zod error details
    - On DB error, return 500 with generic message

    Follow existing API patterns (see `server/api/businesses/index.post.ts` for reference). Use Nitro auto-imports (no explicit imports for `createError`, `readBody`, `defineEventHandler`).

    Import only what's needed:
